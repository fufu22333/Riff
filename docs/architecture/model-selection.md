# 模型选型

## 推荐方案

优先使用 OpenAI 作为第一版实现供应商：

- 多模态对话：`gpt-5.4-mini`
- 更低成本的纯文本降级或摘要：`gpt-5.4-mini`
- ASR：`gpt-4o-mini-transcribe`
- P0 阶段 TTS：浏览器 `SpeechSynthesis`
- P0 稳定后的 TTS：供应商 TTS API

## 为什么这是第一版的最佳选择

Riff 需要一次模型调用同时理解图像，并输出结构化 JSON。P0 的目标不是使用绝对最强的视觉推理模型，而是做出一个稳定的演示：摄像头截图必须明确影响 AI 回复。

OpenAI 是最简单的第一选择，因为它可以覆盖：

- 图像 + 文本理解；
- 结构化 JSON 输出；
- 语音转文本；
- 后续文本转语音；
- 用于摘要和降级的小型文本模型。

本项目不要把 `gpt-4o` 作为主要多模态聊天模型。当前 OpenAI 模型文档已将新项目指向 GPT-5.5 / GPT-5.4 系列，弃用文档中也列出了旧 GPT-4o 快照的关停计划。对这次比赛来说，`gpt-5.4-mini` 是务实默认值，因为它支持图像输入、结构化响应，并且相比旗舰模型有更低的延迟和成本。

Claude 也擅长视觉推理和中文解释，但它不能以同样方式简化 ASR/TTS。对短周期比赛项目来说，减少供应商数量比挤出一点质量增益更重要。

## ASR 与 TTS 说明

ASR 指自动语音识别：把用户语音转换成文本。

在 Riff 中：

```text
microphone audio -> ASR -> userText -> /api/chat
```

TTS 指文本转语音：把 AI 文本转换成可播放的语音音频。

在 Riff 中：

```text
replyText -> TTS -> playable voice audio
```

它们是彼此独立的任务。AI 视觉模型不会自动听到麦克风音频，也不会自动发声。浏览器负责录音，ASR 将语音转换为文本，`/api/chat` 完成理解并回复，TTS 可选地朗读回复。

## P0 策略

P0 阶段按以下方式实现：

1. 实现真实 ASR，因为语音输入是核心交互的一部分。
2. 使用浏览器 `SpeechSynthesis` 作为 TTS 降级方案，因为它快速且免费。
3. 在架构中保留 `/api/tts`，但不要让专业 TTS 阻塞 P0。
4. 等摄像头、ASR、`/api/chat` 和七牛云链路稳定后，再接入供应商 TTS。

如果所选账号无法使用 `gpt-4o-mini-transcribe`，使用 `whisper-1` 作为兼容降级。

这样可以避免项目走偏：花一整天打磨语音播放，但视觉对话核心仍不稳定。
