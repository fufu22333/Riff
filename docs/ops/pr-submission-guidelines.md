# PR 提交规范与作品有效性

本文记录比赛提交对 PR 和 commit 的要求，作为 Riff 后续开发、提交和复核的本地依据。

## 作品有效性

- 作品需符合所选议题方向，并确保自主完成、无抄袭。
- 开发全周期需保持持续交付，严禁临尾突击提交。
- 从议题发布之日起，开发周期内应保持持续的 Pull Request 记录和 commit 提交。
- 仅在最后一天一次性导入所有代码的作品，将直接视为无效作品。
- 所有 commit 时间戳必须落在所选批次的开始与截止时间之内，否则视为无效作品。

以下情形也会导致作品无效或被质疑：

- PR 描述空白，或与实际代码变更严重不符。
- 引用了第三方库或框架，但未在 README 中列明依赖，也未说明原创功能部分。
- 复用了自己过去的代码片段，但未在 PR 描述中注明来源。

## PR 提交规范

- 基于 PR 添加新功能。
- 每个 PR 只做一件事：一个 PR 只实现或修改单一功能。
- 鼓励尽可能小、粒度尽可能细的 PR。
- 大功能应拆分为多个独立 PR 分步提交。
- PR 合并后，主分支代码需保持可运行状态；评委在任意时间查看时应能复现演示效果。

## PR 标题与描述

每个 PR 必须包含清晰完整的标题和描述：

- 标题：一句话说明本 PR 新增或修改了什么。
- 功能描述：说明该功能的作用与使用方式。
- 实现思路：简要说明技术选型或核心实现逻辑。
- 测试方式：说明如何验证该功能正常运行。

## Riff 当前执行口径

- 使用 stacked PR 链路推进，后一个 PR 以前一个 PR 分支为 base。
- PR0 到 PR5 依次交付工程骨架、摄像头、语音 ASR、多模态 chat、对话主链路、七牛存储证据。
- PR 描述必须写明功能、实现、测试和已知边界。
- README 必须持续列明主要第三方依赖和原创功能范围。
- 本地规划文档保持 ignored，不进入公开 main；只提交与 PR 交付直接相关的安全 ops 说明。
