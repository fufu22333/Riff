# Riff 设计文档

## 题目理解

本项目选择七牛云 XEngineer 挑战赛题目一：AI 视觉对话助手。题目要求应用能够打开摄像头与麦克风，让 AI 结合摄像头中的视觉内容和用户说的话进行回应，同时需要考虑视觉理解准确性、语音交互自然度与流畅性，以及端云协同下的运营成本控制。

Riff 的产品定位是“面向音乐创作者的视觉对话助手”。用户在桌面、乐器、歌词纸、房间氛围或临时灵感场景前说话，Riff 结合当前画面与语音意图，输出自然回复、视觉依据和音乐创作方向。

## 计划实现的用户故事

1. 作为用户，我希望打开摄像头后能看到实时预览，确认 AI 当前可以观察的画面范围。
2. 作为用户，我希望通过麦克风直接说出想法，而不是手动输入文字。
3. 作为用户，我希望 AI 回答时说明它看到了什么，避免在视觉不可用时编造画面。
4. 作为音乐创作者，我希望 AI 能把画面氛围和我的口头描述转成情绪、速度、配器、结构和生成提示词。
5. 作为演示和评审场景下的使用者，我希望关键对话轮次、截图和产物能被保存，方便复盘和证明链路真实发生过。
6. 作为用户，我希望 AI 的文字回复可以被朗读，减少一直看屏幕的负担。
7. 作为用户，我希望能听到一个参考音频，快速感受当前创作方向。

## 最终实现情况

已实现：

- 摄像头实时预览、权限失败提示、关闭摄像头，以及每轮对话前自动截取当前帧。
- 麦克风录音、简单音量 VAD、停顿后自动提交，以及手动停止提交。
- ASR Provider 抽象，支持 fake provider 和 OpenAI-compatible ASR provider。
- 多模态对话 Provider 抽象，支持 fake provider 和 OpenAI-compatible vision chat provider。
- 每轮请求包含用户语音转写文本、当前帧截图、最近对话摘要。
- AI 回复采用结构化 JSON，包括 `replyText`、`visualObservation`、`musicSuggestion`、`followUpQuestion` 和 `suggestedActions`。
- 视觉不可用时明确展示原因，例如无摄像头权限、截图失败、无可用视觉主体或视觉接口失败。
- 七牛云对象存储 Provider，可保存截图、单轮 JSON、会话 JSON 和参考音频证据。
- 浏览器 TTS 兜底朗读；服务端 TTS 保留了异步任务接口。
- 参考音频播放链路，包括生成按钮、异步任务状态、fallback sample 和存储证据。
- 单元测试、API 测试和 Playwright smoke 测试覆盖摄像头、语音、对话、存储和参考音频主链路。

未完整实现或保守实现：

- 当前视觉理解采用“每轮当前帧截图”，不是连续视频流理解。这样做可以显著降低 token、带宽和调用成本，也减少实时视频接入的不稳定性。它能覆盖大多数静态场景和演示场景，但对快速动作、连续手势和动态事件理解有限。
- 当前 TTS 主要使用浏览器朗读兜底，没有接入完整云端 TTS 音色服务。
- 当前参考音频不是可商用的真实音乐生成。项目保留了 `/api/generate` 异步接口和证据存储，但实际输出是 fake/fallback reference-only 音频，用于表达产品闭环和交互意图。
- 真实语音开麦场景需要在安静环境中补充人工演示测试。自动化 smoke 已覆盖浏览器麦克风 API 和 ASR 提交流程，但真人口音、噪声和设备差异仍需要现场验证。

## 核心交互流程

```text
打开摄像头 -> 实时预览 -> 用户点击开始说话
-> 浏览器录音与 VAD -> ASR 转写
-> 对话提交前截取当前帧 -> 多模态 AI 回复
-> 展示视觉依据、音乐建议、追问和操作
-> 可选朗读 / 参考音频播放
-> 七牛云保存截图、JSON 和音频证据
```

这个流程优先保证每一轮对话都有清晰证据：用户说了什么、AI 基于哪一帧画面判断、返回了哪些结构化字段、是否有云端记录。

## 视觉理解设计

视觉侧没有把完整视频流持续上传给模型，而是在用户完成一轮语音后立即截取一张当前帧。原因有三点：

- 用户意图通常在一句话内完成，当前帧足以提供桌面、乐器、物体、光线和氛围等上下文。
- 图片压缩到合理尺寸后，传输和模型调用成本明显低于连续视频。
- 静态图片更容易做失败处理：无权限、截图失败、过暗、模糊、无主体时都能明确提示，不需要编造画面。

为了降低误判，Prompt 和 Schema 要求 AI 在视觉不可用时设置 `visualObservation.isUsable=false`，并填写 `failureReason`；如果视觉可用，`summary` 只能描述截图中能看见的内容。

## 语音交互设计

语音侧采用浏览器原生 `getUserMedia` 获取麦克风音频，通过 `AudioContext` 读取 PCM 数据并编码为 WAV，再提交给 `/api/asr`。

交互上提供两种方式：

- 自动方式：VAD 检测到用户说话后，等待静音超过阈值自动提交。
- 手动方式：用户点击“停止”立即提交当前录音。

这样做可以兼顾自然度和可控性。自动提交减少用户操作，手动停止便于演示时掌控节奏。当前 VAD 采用音量阈值和静音计时，属于轻量实现，优势是端侧成本低，缺点是对强噪声环境不够鲁棒。

## 端云协同与成本控制

计划考虑的成本控制技巧：

1. 端侧先做摄像头预览和截图压缩，避免持续上传视频。
2. 只有用户完成一轮语音后才调用 ASR 和多模态模型，避免空转调用。
3. 视觉图片使用低细节模式和较小尺寸，优先满足语义理解而不是高清识别。
4. 保留最近几轮对话摘要，而不是发送完整长历史。
5. 所有外部能力都通过 Provider 抽象接入，开发和测试时可以用 fake provider。
6. 视觉、ASR、TTS、音乐生成失败时有本地或文本兜底，避免单点失败导致整轮体验中断。
7. 七牛云只保存关键证据产物：截图、单轮 JSON、会话 JSON 和参考音频，不保存持续视频流。

实际采用的策略：

- 当前帧截图代替视频流上传。
- 截图尺寸压缩和图片格式选择。
- ASR、Chat、Storage、TTS、Music Provider 抽象。
- fake provider 用于自动化测试和低成本演示环境。
- 对话历史压缩为最近几轮摘要。
- 失败原因结构化，避免重复无效重试和编造输出。
- 参考音频使用 reference-only fallback，避免把高成本音乐生成作为主链路依赖。

## 架构说明

前端：

- `components/camera/CameraPreview.tsx`：摄像头权限、实时预览、截图和视觉失败状态。
- `components/recorder/VoiceRecorder.tsx`：麦克风权限、录音、VAD、WAV 编码和 ASR 提交。
- `app/page.tsx`：串联摄像头、语音、对话、TTS、参考音频和云端证据展示。

后端 API：

- `POST /api/asr`：语音转文字。
- `POST /api/chat`：多模态对话主链路。
- `POST /api/tts`、`GET /api/tts/{jobId}`：异步朗读接口。
- `POST /api/generate`、`GET /api/generate/{jobId}`：参考音频任务接口。
- `GET /api/generate/sample`：返回可播放 fallback 样例。

服务端 Provider：

- `lib/server/asr/*`：ASR provider。
- `lib/server/ai/*`：多模态 chat provider。
- `lib/server/storage/*`：fake storage 和七牛云对象存储。
- `lib/server/tts/*`：TTS 任务接口与浏览器 fallback。
- `lib/server/music/*`：参考音频任务与 fallback sample。

当前自动化验证覆盖：

- 单元测试：摄像头截图、VAD、Recorder、视觉证据、音乐建议、合约 Schema。

```bash

npm run test:smoke
```

## 已知边界与后续方向

后续最值得继续投入的方向：

1. 补充真人开麦测试和 demo 视频片段，覆盖普通话、英文短句和中英混合表达。
2. 将单帧截图升级为低频多帧采样，例如每轮取开始、中间、结束三帧，并生成简单 motion summary。
3. 接入真实 TTS 服务，提供更稳定的 AI 语音回应。
4. 接入真实音乐生成或检索式参考曲库，并在 README 中明确版权和使用范围。
5. 优化 VAD 的噪声适应能力，减少误触发和过早截断。

当前版本的目标是把题目一的主链路打通并真实呈现：摄像头可用、麦克风可用、AI 能结合语音和当前画面回应，失败时不编造，关键证据可追溯。
